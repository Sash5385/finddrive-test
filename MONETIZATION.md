# FindDrive — План монетизації

> Робочий документ. Будуємо фундамент **поступово**, без раптового вмикання оплати.
> Статус кожного кроку позначено: ✅ зроблено · 🔨 в роботі · ⏳ заплановано.

---

## 1. Модель монетизації (зафіксована)

- **Підписка для інструкторів.** Не комісія із занять.
- **Оплата за заняття — напряму між учнем та інструктором.** Сервіс у розрахунках не бере участі.
- **Для учнів пошук і запис — завжди безкоштовні.**
- Модель уже закріплена в **Умовах партнерства** (`index.html`, розділ «Записи, оплата та проведення занять» + «Оплата»): комісії немає, платна підписка з'явиться пізніше, **із заблаговременним повідомленням**, без списань заднім числом.

> Наслідок для реалізації: ми **не** інтегруємо split-платежі / ескроу / комісію. Потрібен лише механізм **регулярної підписки інструктора** + прапорець «підписка активна» на його профілі.

---

## 2. Тригер запуску (коли вмикати оплату)

Не за датою, а за **критичною масою** — коли на платформі достатньо активних інструкторів, щоб підписка мала цінність (вхідний потік учнів).

- Метрика: **кількість активних (approved) інструкторів**, окремо — **верифікованих**.
- Орієнтовний поріг для першого міста (Київ): **~30 активних інструкторів** з реальним потоком записів. Поріг — параметр, не константа в коді назавжди.
- **Крок 1 фундаменту (✅):** датчик критичної маси в адмінці — бачимо прогрес до порога, ще нічого не вмикаючи.

---

## 3. Що дає підписка (ціннісна пропозиція)

Підписка не має «ламати» безкоштовний сценарій — інакше відлякаємо ранніх інструкторів. Вона дає **підсилення видимості**, а не доступ до базового функціоналу.

| Перк | Опис | Технічно |
|------|------|----------|
| **Бейдж «Верифіковано»** | Синій щит на картці й у профілі | Уже є (`verified`), прив'язати до активної підписки |
| **Пріоритет у видачі** | Вище в списку інструкторів на головній | Сортування `#instr-grid` за `subscription.active` |
| **Мітка на карті помітніша** | Виділений пін | Опційно, пізніше |
| **Статистика профілю** | Скільки переглядів / кліків «Записатись» | Потребує лічильників (пізніше) |
| **Більше точок зустрічі / фото** | М'який ліміт для безкоштовних | Пізніше, обережно |

**Базове (завжди безкоштовно):** бути в каталозі, приймати записи, чат, календар. Ніколи не ховаємо це за paywall для вже зареєстрованих — лише заблаговременно попереджаємо про майбутні зміни.

---

## 4. Тарифи (чернетка, не фінал)

- **Один тариф** на старті (простіше тримати обіцянку з ToS): напр. **199–299 грн/міс** за інструктора.
- **Річна знижка**: −2 місяці при оплаті за рік.
- **Інтеграція з ID4Drive**: спільна підписка дешевша, ніж дві окремі (див. §9).
- Перші N інструкторів або перші X місяців — **безкоштовний період** («early adopter»), щоб не втратити ранню базу.

---

## 5. Платіжний провайдер

Рекомендація: **LiqPay (Приват)** або **WayForPay** — обидва підтримують **регулярні списання (subscription/recurring)** у гривні й поширені серед українських ФОП. Stripe/Paddle — гірше для UAH-рекурингу та виплат в Україну.

- **LiqPay**: `subscribe` API, callback-сервер (server-to-server), підпис `sha1(private_key + data + private_key)`.
- **WayForPay**: `REGULAR` платежі, `serviceUrl` webhook.

Рішення — після узгодження ФОП/рахунку. Обидва вимагають **серверної частини** (перевірка підпису callback) — вже маємо Cloud Functions.

---

## 6. Модель даних (Firestore)

Розширюємо документ `instructors/{id}` полем-підоб'єктом (спочатку **опційним**, дефолт = «безкоштовно/активно», щоб нічого не зламати):

```js
subscription: {
  status: 'free' | 'trialing' | 'active' | 'past_due' | 'canceled',
  plan:   'monthly' | 'yearly' | null,
  provider: 'liqpay' | 'wayforpay' | null,
  providerRef: string | null,   // id підписки у провайдера
  currentPeriodEnd: Timestamp | null,
  startedAt: Timestamp | null,
}
```

- Правило видимості: інструктор **видимий у каталозі**, якщо `status == 'approved'` **І** (`subscription` відсутня **АБО** `subscription.status ∈ {free, trialing, active}`). Тобто до запуску оплати всі — `free` → нічого не змінюється.
- **Пише `subscription` лише сервер** (Cloud Function через Admin SDK) — інструктор не може сам собі поставити `active`. У `firestore.rules` для інструктора це поле — read-only (додати до заборонених до self-запису ключів).

---

## 7. Зміни у Firestore Rules (⏳ на етапі запуску, не зараз)

- `instructors` update інструктором: заборонити зміну ключа `subscription` (як зараз заборонено `status`).
- Нова колекція `payments/{id}` (лог транзакцій): `allow read: if isAdmin() || власник`; `write: if false` (лише сервер).
- Callback-и провайдера пишуться через Admin SDK, повз rules.

---

## 8. Cloud Functions (⏳ на етапі запуску)

- `paymentCallback` (`onRequest`) — приймає webhook LiqPay/WayForPay, перевіряє підпис, оновлює `instructors/{id}.subscription`, пише `payments`.
- `subscriptionExpiryCron` (`onSchedule`, щодня) — переводить прострочені `active → past_due → canceled`, шле push-нагадування «підписка закінчується».
- (Уже є `sendPush` — перевикористати для нагадувань про оплату.)

---

## 9. Інтеграція з ID4Drive (⏳)

- Спільна підписка: інструктор, що вже платить в ID4Drive, отримує знижку/безкоштовний доступ до FindDrive (і навпаки).
- Технічно: спільний ідентифікатор інструктора або звірка за email/телефоном між проектами; єдиний біллінг-провайдер спрощує.
- Деталі узгоджуються з боку ID4Drive окремо.

---

## 10. UI (⏳ на етапі запуску)

- **Кабінет інструктора**: секція «Підписка» — статус, дата продовження, кнопка «Оформити / Продовжити», історія платежів.
- **Paywall — м'який**: не блокуємо базу; показуємо банер «Отримайте пріоритет у видачі» + переваги. Жорсткий paywall (приховування з каталогу) — тільки після офіційного заблаговременного повідомлення й лише для нових, згідно з ToS.
- Усі вікна/модалки — за дизайн-системою (обводка `1px solid #000`, палітра `--green`, header зелений) — див. `CLAUDE.md`.

---

## 11. Юридичне

- ToS уже містить застереження про майбутню підписку — **зберегти обіцянку**: перед запуском окреме повідомлення (email/банер/push) за розумний строк.
- Додати **Публічну оферту** для платної підписки + реквізити ФОП перед першим списанням.
- Без списань заднім числом за вже надані безкоштовні послуги.

---

## 12. Дорожня карта (поетапно, «потихоньку»)

| Фаза | Що | Ризик | Статус |
|------|-----|-------|--------|
| **0. Фундамент** | Датчик критичної маси в адмінці; цей документ | Нульовий (read-only) | ✅ |
| 0.1 | Модель `subscription` як опційне поле + дефолт `free` у логіці видачі (нічого не вмикає) | Низький | ⏳ |
| 0.2 | Сортування видачі: активна підписка → вище (поки активних немає — порядок не змінюється) | Низький | ⏳ |
| 1. Оферта/ФОП | Юр. підготовка, вибір провайдера (LiqPay vs WayForPay) | Поза кодом | ⏳ |
| 2. Біллінг backend | `paymentCallback`, `subscriptionExpiryCron`, `payments`, rules | Середній | ⏳ |
| 3. Біллінг UI | Секція «Підписка» в кабінеті, м'який банер | Середній | ⏳ |
| 4. Запуск | Заблаговременне повідомлення → увімкнення тарифу | Високий (комунікація) | ⏳ |
| 5. ID4Drive | Спільна підписка | Залежить від ID4Drive | ⏳ |

**Принцип:** кожен крок до Фази 4 **не змінює** поведінку для чинних інструкторів і учнів. Ми будуємо рейки, поїзд не рушає без окремого рішення й попередження.
